# Ctrl 고정 현상 조사 기록 (2026-04-22)

## 요약
- 사용자가 체감한 대표 패턴은 "터미널에서 AI가 빌드/테스트를 실행한 뒤 Ctrl 키가 눌린 채 남는 것 같음"이었다.
- 추가로 음성 전사 앱 단축키(`Ctrl+Space`) 사용 뒤에도 같은 현상이 있었다.
- 조사 결과 AltKey에는 modifier를 실제 OS 키 상태로 유지하는 구조적 취약점이 있었지만, 이번 재현 로그와 수동 검증만으로는 AltKey가 주원인이라고 확인되지는 않았다.

## 확인된 사실
- `ToggleModifier(VK_CONTROL)`는 실제로 `SendKeyDown(VK_CONTROL)`를 보낸다.
- 숨김 전까지 해제가 되지 않으면 OSK에서 Ctrl이 눌린 상태로 보일 수 있는 구조였다.
- 이에 따라 다음 방어 수정은 유지한다.
  - 트레이로 숨길 때 `Ctrl/Alt/Win` 같은 고위험 modifier 해제
  - 키보드 창 닫기 버튼으로 숨길 때 고위험 modifier 해제
  - 앱 종료 시 전체 modifier 해제

## 재현 결과
- AltKey 실행 중 `Ctrl`을 누른 뒤 트레이로 숨기면 Ctrl이 정상 해제됨을 확인했다.
- 최소화는 숨김이 아니므로 modifier 상태가 유지되는 현재 동작을 정상으로 간주했다.
- 키보드가 켜진 상태에서 받아쓰기를 사용했을 때 특이사항은 확인되지 않았다.
- AI가 `dotnet build`를 실행한 직후에는 Ctrl 고정 현상이 재현되지 않았고, 당시 AltKey 내부 상태도 비어 있었다.

## 해석
- AltKey는 "Ctrl 고정을 만들 수 있는 취약한 경로"가 있었고, 숨김/종료 경로는 방어가 추가되었다.
- 그러나 사용자가 가장 강하게 느낀 "빌드 직후 Ctrl 고정"은 이번 점검에서 AltKey 쪽 증거가 나오지 않았다.
- 따라서 현재 시점에서는 AltKey 단독 원인보다는 다음 가능성이 더 크다.
  - 음성 전사 앱의 전역 단축키 처리
  - 다른 입력 보조 도구 또는 가상키보드
  - Windows 입력 상태 자체의 일시적 꼬임

## 후속 대응
- 이번 조사용 텍스트 로그 계측은 제거했다.
- 동일 증상이 다시 발생하면 다음을 함께 확인한다.
  - Windows OSK에서 실제로 왼쪽 Ctrl이 눌려 있는지
  - 당시 AltKey가 트레이 숨김/종료 직전이었는지
  - 음성 전사 앱 또는 다른 입력 도구를 방금 사용했는지

## 결론
- AltKey의 숨김/종료 관련 modifier 방어는 유지한다.
- 이번 시점에는 AltKey를 Ctrl 고정의 주원인으로 단정하지 않는다.
