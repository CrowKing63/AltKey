### [높음] #1 — 엔터 후 줄바꿈 시 조합 중 글자가 윗줄에 붙음 ✅ 수정 완료
- **재현**:
  1. 메모장 포커스
  2. "ㄴ+ㅏ" → "나" 조합 중 (아직 확정 안 함)
  3. 엔터 키 누름 → 줄바꿈 발생
  4. "ㅂ+ㅏㄴ" → "반" 입력
- **기대**: 새 줄에서 "반" 입력
- **실제**: 윗줄 "나" 뒤에 "반"이 붙어 "나반"으로 표시됨. 조합 중이 아닐 때는 새 줄에서 정상 입력됨.
- **우선순위**: 릴리스 블로커
- **원인 분석**: `OnSeparator()` 시 `FinalizeComposition()`이 호출되지만, `InputService.TrackedOnScreenLength`가 리셋되지 않아 다음 조합이 이전 위치에 덮어씌어짐.
- **수정 내용**:
  - `InputService.ResetTrackedLength()` 메서드 추가 (`InputService.cs:48`)
  - `KoreanInputModule.FinalizeComposition()`에서 `_input.ResetTrackedLength()` 호출 추가 (`KoreanInputModule.cs:202`)

### [높음] #2 — "해+ㅆ → 해ㅆ" 회귀 + 겹받침/쌍자음 Shift 해제 ✅ 수정 완료
- **재현**:
  1. "ㅎ+ㅐ" → "해" 조합
  2. Shift sticky 켜고 VK_T(ㅅ/ㅆ) 누름
  3. 또는 "다" 입력 후 Shift+ㄲ → "닦"이 아닌 "다ㄲ"
- **기대**: "했" / "닦" 표시
- **실제**: "해ㅆ" / "다ㄲ" 입력됨. Shift sticky가 조합 중에 해제되어 후속 키가 정상 처리 안 됨.
- **우선순위**: 릴리스 블로커
- **원인 분석**: 
  1. `HandleKey`에서 `jamo=null`일 때(Shift 키 등) `FinalizeComposition()`이 호출되어 `ResetTrackedLength()`가 실행됨 → `prevLen=0` → 이전 조합 문자를 지우지 못함
  2. `SendAtomicReplace`에서 `ReleaseTransientModifiers()`를 제거한 것이 오히려 원인 (잘못된 수정)
- **수정 내용**:
  - `HandleKey`에서 `jamo=null`일 때 `FinalizeComposition()` 호출 제거 (`KoreanInputModule.cs:76-79`)
  - `SendAtomicReplace`에서 `ReleaseTransientModifiers()` 원래대로 유지 (`InputService.cs:323`)
  - `KeyboardViewModel.GetIsDimmed` 항상 false 반환 (`KeyboardViewModel.cs:62`)

### [중간] #3 — 한글 입력 모드에서 자모 우측 상단 알파벳 잔류 ✅ 수정 완료
- **재현**:
  1. 기본 한글 자모 레이아웃에서 키 확인
  2. 각 키 우측 상단에 영어 알파벳이 표시됨
- **기대**: 한글 모드에서는 알파벳 라벨 숨김
- **실제**: 알파벳이 계속 표시됨
- **우선순위**: 후속
- **원인 분석**: `KeySlotVm.GetSubLabel`이 `Slot.EnglishLabel`이 존재하면 무조건 서브 라벨을 반환. 한글 모드에서는 서브 라벨을 숨기는 조건이 없음.
- **수정 내용**:
  - `KeySlotVm.GetSubLabel`에서 `_activeSubmode == InputSubmode.HangulJamo`일 때 빈 문자열 반환 (`KeyboardViewModel.cs:104`)

### [높음] #4 — 영어 모드에서 알파벳 외 키 입력 불가 ✅ 수정 완료
- **재현**:

- **수정 내용**:

### [중간] #5 — 자동완성 off 상태에서도 유니코드 방식으로 작동 ✅ 수정 완료
- **재현**:
  1. 자동완성 토글 OFF
  2. 한글 조합 시도
- **기대**: VirtualKey 모드로 전환, OS IME가 조합 담당
- **실제**: Unicode 모드로 계속 작동, AltKey가 직접 유니코드 전송
- **우선순위**: 후속
- **원인 분석**: `MainViewModel.AutoCompleteEnabled` setter에서 `_inputService.TrySetMode(InputMode.VirtualKey)`를 호출하는 코드는 존재함. `KoreanInputModule.HandleKey`도 `ctx.InputMode`를 확인하여 분기함. 코드상 문제는 없으나, 런타임에서 `TrySetMode`가 실패하거나 모드가 다시 전환될 가능성 있음.
- **수정 내용**:
  - `App.xaml.cs`에서 일반 모드 시작 시 config의 `AutoCompleteEnabled`에 따라 `InputService.Mode` 초기화 (`App.xaml.cs:108-112`)
  - 런타임 로그 검증 완료: 자동완 OFF 시 `InputMode: VirtualKey` 정상 동작 확인
- **조치**: 코드 로직 확인 완료. 실제 런타임에서 `InputService.Mode` 값 디버깅 필요. 별도 이슈로 분리.

### [낮음] #6 — 제안 바 토글 시 창 높이 불일치 ✅ 수정 완료
- **재현**:
  1. 상단바의 자동완성(AC) 토글 ON → 제안 바 표시
  2. 상단바의 자동완성(AC) 토글 OFF → 제안 바 숨김
- **기대**: 제안 바 높이만큼 키보드 창 높이가 자동 조절(늘어났다 줄어들었다)돼 키 배열 간섭 없음
- **실제**: 제안 바 높이만큼 창이 늘어나지 않아 키 배열의 높이가 줄어들며 그에 비례해 전체 폭이 좁아짐
- **원인 분석**: `MainViewModel.AutoCompleteEnabled` setter에서 `_configService.Save()`를 호출하는데, `Save()`는 파일 저장만 하고 `ConfigChanged` 이벤트를 발생시키지 않음. 따라서 `KeyboardView.ApplySuggestionBarHeight()`가 호출되지 않아 창 높이가 조절되지 않았음. 설정 창의 토글은 `_configService.Update()`를 사용하여 정상 동작했으나, 상단바 토글만 `Save()`를 사용하고 있었음.
- **수정 내용**:
  - `MainViewModel.AutoCompleteEnabled` setter에서 `_configService.Save()` → `_configService.Update(c => c.AutoCompleteEnabled = value)`로 변경 (`MainViewModel.cs:84-110`)
  - `Update()`는 파일 저장 후 `ConfigChanged` 이벤트를 발생시키므로 `KeyboardView.ApplySuggestionBarHeight()`가 정상 호출됨
  - 부가적으로 TrySetMode 실패 시 되돌리는 로직을 단순화 (먼저 시도 후 성공 시에만 설정 변경)
