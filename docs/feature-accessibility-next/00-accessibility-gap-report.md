# 접근성 기능 보강 보고서

## 결론 요약

현재 접근성 기능은 1차 토대가 들어갔지만, 중증 운동장애 사용자가 실제로 매일 쓰기에는 부족하다. 특히 스위치 제어는 "자동으로 키를 훑고 Enter/Space로 고른다" 수준이라 Apple 스위치 제어, Android 스위치 제어가 제공하는 사용자 입력 장치 매핑, 스캔 범위 선택, 제안/메뉴/시스템 제어 확장성에 비해 기능 폭이 좁다.

가장 큰 실사용 격차는 다음 네 가지다.

1. 스위치 입력 장치가 `Tab`, `Space`, `Enter`에 고정되어 있다.
2. 스캔 대상이 키보드 본체 키에만 묶여 있고 자동완성 제안 바를 선택할 수 없다.
3. 스캔 방식이 단일 순차 스캔 중심이라 행/열 스캔, 그룹 스캔, 수동 스캔 같은 선택권이 없다.
4. 탭 네비게이션과 스위치 제어가 같은 하이라이트 상태를 공유하지만 사용자 입장에서는 모드, 범위, 현재 위치, 탈출 방법이 충분히 명확하지 않다.

## 현재 구현에서 확인한 내용

- `AccessibilityNavigationService`는 저수준 키보드 훅으로 `Tab`, `Enter`, `Space`를 감지한다.
- `SwitchScanEnabled`가 켜져 있으면 스위치 모드가 탭 네비게이션보다 우선한다.
- 2스위치 모드는 `Tab = 다음`, `Enter/Space = 선택`으로 고정되어 있다.
- `KeyboardViewModel`은 `_a11yNavigableSlots`에 키보드 본체의 `KeySlotVm`만 담는다.
- `SuggestionBar`의 현재 단어, 제안 단어, 삭제 메뉴는 일반 WPF 버튼이지만 스캔 대상 모델에는 포함되지 않는다.
- 스캔 타이머는 `SwitchScanIntervalMs` 하나만 사용하며, 시작 지연·첫 항목 대기·선택 후 일시정지·반복 횟수 제한 같은 세밀한 조정이 없다.
- `LiveRegionService`는 공지 토대만 있고, 공지 빈도 제한이나 공지 종류 설정은 없다.
- 테스트는 `AccessibilitySafetyTests`에 기본값과 일부 UI 속성만 있어서 스캔 동작 회귀를 잡기 어렵다.

## 기능별 미흡점

### 스위치 제어

현재 스위치 제어는 외부 스위치가 키보드의 특정 키를 보내는 경우에만 쓸 수 있다. 실제 보조공학 장치는 단일 버튼, 게임패드 버튼, HID 스위치 인터페이스, 발판, sip-and-puff 장치처럼 다양한 입력을 보낼 수 있으므로 사용자 매핑이 필요하다.

스캔 방식도 부족하다. 키가 많을수록 단일 순차 스캔은 원하는 키까지 기다리는 시간이 길어진다. 한국어 입력은 글자 수가 많고 자동완성 활용 가치가 크므로, 행/열 스캔과 자동완성 우선 스캔이 중요하다.

### 자동완성 접근성

자동완성은 AltKey의 핵심 기능인데, 스위치 제어 사용자는 현재 제안을 스캔으로 선택할 수 없다. 자동완성 제안을 고를 수 없으면 한글 입력 부담을 크게 줄이는 핵심 접근성 효과가 사라진다.

단, 자동완성 내부 조합 로직을 직접 수정하면 안 된다. 해결 방향은 제안 바 UI와 `SuggestionBarViewModel`의 명령을 스캔 대상에 노출하는 것이다.

### 탭 네비게이션

현재 탭 네비게이션은 키보드 본체 키 이동과 선택에는 도움이 된다. 그러나 헤더 버튼, 자동완성 바, 설정 버튼, 닫기/접기 같은 보조 기능을 어떻게 접근할지 선택권이 부족하다. 또한 `Tab`을 AltKey 내부 탐색으로 가로채는 모드에서는 사용자가 대상 앱으로 `Tab`을 보내고 싶을 때 혼란이 생길 수 있다.

### 시각/음성 피드백

하이라이트와 LiveRegion 공지는 있으나, 스캔이 빠를 때 낭독이 과도해질 수 있다. 반대로 TTS가 꺼져 있으면 저시력 사용자는 현재 위치를 놓치기 쉽다. 공지 방식은 "항상 읽기"가 아니라 사용자 선택 가능해야 한다.

### 설정 UI

접근성 설정이 오른쪽 컬럼에 들어가 있지만, 스위치 제어가 실사용 수준이 되려면 설정 항목이 훨씬 많아진다. 지금 구조에 단순 체크박스를 계속 추가하면 설정이 길고 어려워진다. 별도 그룹, 고급 설정 접기, 입력 키 캡처 버튼이 필요하다.

## 개선 원칙

- 입력 안정성 우선: 스캔은 기존 `KeyPressedCommand`, `AcceptSuggestionCommand`, `CommitCurrentWordCommand`를 호출해야 한다.
- 자동완성 보호: `KoreanInputModule`과 `HangulComposer`를 직접 건드리지 않는다.
- 작게 나누기: 먼저 입력 장치 커스텀, 다음 스캔 엔진, 그다음 자동완성 바를 연결한다.
- 사용자 통제권: 속도, 방식, 입력 키, 공지, 범위를 설정할 수 있어야 한다.
- 되돌릴 수 있음: 새 옵션은 꺼짐 기본값 또는 기존 동작과 같은 기본값으로 시작한다.

## 우선순위

1. 스위치 입력 장치 커스텀
2. 자동완성 제안 바 스캔 대상 포함
3. 행/열 스캔과 선택 후 일시정지
4. 탭 네비게이션 범위 선택과 탈출 동작
5. 공지/TTS 빈도 제어와 테스트 보강
