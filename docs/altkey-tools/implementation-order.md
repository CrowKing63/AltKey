# 구현 직전 작업 순서: AltKey.Tools 1단계

## 목적

이 문서는 실제 구현에 들어갈 때 작업 순서를 흔들리지 않게 고정하기 위한 것이다.

## 권장 순서

### 1. 프로젝트 뼈대 생성

먼저 한다.

- `AltKey.Tools` 프로젝트 생성
- WPF 설정
- 최소 의존성 패키지 추가
- 도구 호스트 창 추가

이 단계의 목표는 "컴파일되는 빈 도구 앱"이다.

### 2. 레이아웃 편집기 먼저 이관

두 편집기 중 레이아웃 편집기를 먼저 옮긴다.

이유:

- 의존성이 더 단순하다.
- 파일 기반 경계가 명확하다.
- 저장 후 메인 재로드 모델을 먼저 검증하기 좋다.

완료 조건:

- 도구 앱에서 레이아웃 편집기가 열린다.
- 저장이 된다.
- 메인 앱이 재로드 후 반영한다.

### 3. 메인 앱 진입점 교체

레이아웃 편집기가 도구 앱에서 정상 동작하면, 메인 앱 설정창에서 기존 직접 생성 경로를 끊는다.

완료 조건:

- 설정창에서 도구 앱을 통해 레이아웃 편집기가 열린다.

### 4. 사용자 단어 편집기 이관

다음으로 사용자 단어 편집기를 옮긴다.

이 단계에서 주의:

- 한국어/영어 탭
- 단어 검색
- 추가/삭제
- bigram 탭
- 종료 시 flush 동작

완료 조건:

- 도구 앱에서 단어 편집기가 열린다.
- 저장/삭제 후 메인 앱 자동완성 데이터 재로드가 된다.

### 5. 접근성/수동 검증

마지막에 몰아서 하지 말고 각 단계 끝마다 확인한다.

- 첫 포커스
- 탭 순서
- `Esc` 닫기
- 스크린리더 읽기
- 메인 앱과 포커스 충돌 여부

## 하지 말아야 할 순서

다음 순서는 피한다.

- 설정창 전체 분리부터 시작
- 레이아웃과 사용자 단어 편집기를 동시에 이관
- 범용 IPC부터 구현
- 자동완성 핵심 구조를 먼저 손보는 방식

## 중간 점검 기준

다음 조건을 만족하면 다음 단계로 넘어간다.

### 레이아웃 편집기 단계 통과 기준

- 도구 앱 단독 실행 가능
- 레이아웃 목록/불러오기/저장 가능
- 메인 앱에서 재로드 후 반영 확인
- 메인 입력 기능 회귀 없음

### 사용자 단어 편집기 단계 통과 기준

- 도구 앱에서 한글 입력 안정적
- 단어/삭제/검색 기능 유지
- bigram 기능 유지
- 자동완성 반영 확인

## 요약

실제 구현은 `프로젝트 생성 -> 레이아웃 편집기 -> 메인 진입점 교체 -> 사용자 단어 편집기 -> 접근성 검증` 순서가 가장 안전하다.
