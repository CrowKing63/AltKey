# 범위 재정의: 설정 앱 전면 분리 대신 편집 도구 앱으로 축소

## 결론

`standalone-settings-app`의 원래 계획은 **지금 바로 실행하지 않는다**.

대신 다음 방향으로 축소한다.

- 메인 `AltKey` 앱은 입력, 자동완성, 접근성 실행, 실시간 설정 반영을 계속 담당한다.
- 별도 프로세스는 `설정 앱`이 아니라 **텍스트 입력이 많은 편집 도구 전용 앱**으로 설계한다.
- 1단계 목표는 `설정 전체 분리`가 아니라 **한글 입력 안정성이 중요한 편집 기능을 메인 입력 프로세스 밖으로 이동**하는 것이다.

이 결정의 핵심 이유는 현재 설정 구조가 단순 UI가 아니라 메인 런타임 제어와 강하게 결합되어 있기 때문이다.

## 원래 계획을 바로 실행하지 않는 이유

### 1. 현재 설정창은 독립 UI가 아니라 런타임 제어 허브에 가깝다

현재 `SettingsViewModel`은 다음 책임을 함께 가진다.

- 설정 저장
- 테마 적용
- 전역 단축키 재등록
- 시작 프로그램 등록 변경
- 사운드 설정 적용
- 업데이트 확인/설치
- 재시작/권한 전환
- 편집기 창 열기

따라서 `SettingsWindow`만 별도 EXE로 떼는 방식은 실제로는 화면 분리가 아니라 **프로세스 경계 재설계**에 가깝다.

### 2. 실시간 반영 구조가 인프로세스 이벤트에 의존한다

현재는 `ConfigService.Update()` 이후 `ConfigChanged` 이벤트를 통해 메인 앱의 여러 컴포넌트가 즉시 반응한다.

- 창 크기
- 상단바 상태
- 접근성 동작
- 스캔/입력 동작
- 제안 바 표시

즉 별도 프로세스가 `config.json`만 수정하는 방식으로는 사용자가 기대하는 "즉시 반영"을 그대로 유지할 수 없다.

### 3. 접근성 리스크가 너무 크다

설정창은 단순 옵션 목록이 아니라 다음 특성을 이미 갖고 있다.

- 초기 포커스 이동
- 탭 전환 시 첫 컨트롤 포커싱
- 포커스 추적기 연동
- 상세 보조 기능 창 호출

전면 분리는 자모 분리 문제를 줄일 수 있지만, 잘못 나누면 키보드 탐색과 스크린리더 사용성이 오히려 나빠질 수 있다.

## 새 기준: 무엇을 메인에 두고 무엇을 별도 프로세스로 둘 것인가

### 메인 프로세스에 남겨야 하는 기능

다음 중 하나라도 만족하면 메인 프로세스 유지가 기본이다.

- 입력 루프를 직접 건드린다.
- 변경 즉시 메인 키보드 상태가 바뀌어야 한다.
- 여러 전역 상태를 동시에 참조한다.
- 접근성 실패 시 사용자가 입력 자체를 못 하게 된다.

예시:

- 자동완성 on/off
- 스위치 스캔
- 포커스 탐색
- TTS 동작
- 전역 단축키
- 창 크기/상단바/항상 위

### 별도 프로세스로 옮기기 좋은 기능

다음 조건에 가까울수록 별도 편집 도구 앱으로 보내기 쉽다.

- 대량 입력, 검색, 정렬, 검수, 일괄 수정이 중심이다.
- 파일 저장이 핵심이다.
- 저장 후 메인 앱이 다시 읽으면 충분하다.
- 실패가 메인 입력 기능을 직접 멈추게 해서는 안 된다.

예시:

- 레이아웃 편집기
- 사용자 단어 편집기
- 프로필 매핑 편집
- 향후 백업/복원 도구
- 향후 CSV/JSON 일괄 편집기

### 혼합이 필요한 기능

다음 유형은 편집은 별도 프로세스에서 하되, 적용은 메인 앱이 담당하는 것이 적절하다.

- 프로필 매핑
- AI 프롬프트/템플릿 편집
- 일부 접근성 상세 설정

## 목표 아키텍처: AltKey.Tools

### 핵심 개념

별도 앱의 이름과 역할은 `설정 앱`보다 **편집 도구 앱**이 더 적절하다.

권장 구조:

- `AltKey`
  - 입력 실행 앱
  - 자동완성/한글 조합/접근성 실행
  - 실시간 설정 반영
- `AltKey.Tools`
  - 도구 선택 화면
  - 레이아웃 편집기
  - 사용자 단어 편집기
  - 이후 텍스트 편집 도구들

### 1단계에서 필요한 최소 요소

- `ToolHost` 메인 창
- 도구 목록 또는 카드형 시작 화면
- `LayoutEditorViewModel` 이관
- `UserDictionaryEditorViewModel` 이관
- 데이터 접근용 얇은 저장소 계층
- 저장 후 메인 앱에 재로드만 알리는 최소 IPC

## 1단계 범위

### 포함

- 별도 `AltKey.Tools` 앱 생성
- 레이아웃 편집기 분리
- 사용자 단어 편집기 분리
- 가능하면 프로필 매핑 편집 분리 검토
- 저장 후 메인 앱 재로드 알림

### 제외

- 설정창 전체 분리
- 범용 IPC 프로토콜
- 설정 전체를 공용 코어로 대이관
- 실시간 양방향 동기화
- 자동완성/한글 조합 핵심 구조 변경
- 대규모 저장 잠금 시스템

## 메인 앱과의 최소 연동 원칙

별도 앱과 메인 앱 사이 통신은 최소화한다.

1단계 권장 명령:

- `ShowTool(toolName)`
- `ReloadLayouts`
- `ReloadUserDictionary`
- `ReloadBigramData`

중요 원칙:

- 레이아웃/사전 데이터 본문을 IPC로 보내지 않는다.
- 양쪽이 같은 데이터 폴더를 읽는다.
- 메인 앱은 "다시 읽기"만 수행한다.

## 구현 우선순위

### 1순위: 레이아웃 편집기

이유:

- 대상 데이터가 명확하다.
- 저장 후 재로드 모델이 단순하다.
- 메인 입력 루프와 직접 결합이 상대적으로 약하다.

### 2순위: 사용자 단어 편집기

이유:

- 한글 입력 안정성 개선 효과가 크다.
- 별도 편집 도구 앱의 가치가 사용자에게 가장 잘 드러난다.
- 다만 자동완성 품질과 연결되므로 검증을 더 엄격히 해야 한다.

### 3순위: 프로필 매핑 편집

이유:

- 데이터 구조는 단순하다.
- 실제 적용은 메인 런타임이 담당하므로 1, 2 이후가 안전하다.

## 접근성 체크포인트

1단계 구현 후 반드시 확인한다.

- 도구 앱 실행 시 첫 포커스 위치가 예측 가능해야 한다.
- 탭 순서가 선형적이고 반복 가능해야 한다.
- 스크린리더가 도구 제목, 주요 입력란, 저장/취소 버튼을 분명히 읽어야 한다.
- 메인 앱과 도구 앱 사이 전환 시 포커스 유실이 없어야 한다.
- 스위치 스캔/키보드 탐색 사용자가 편집 도구를 조작할 수 있어야 한다.
- `ReducedMotionEnabled`를 고려한 불필요한 애니메이션 억제가 가능해야 한다.

## 다음 작업의 기준

앞으로 텍스트 입력이 필요한 새 기능이 생기면 먼저 아래 순서로 판단한다.

1. 실시간 입력 루프를 직접 건드리는가?
2. 변경 즉시 메인 키보드 상태가 바뀌어야 하는가?
3. 여러 전역 상태를 동시에 읽고 쓰는가?
4. 핵심이 저장/검색/검수/일괄 편집인가?
5. 실패가 메인 입력 기능을 방해하면 안 되는가?

판단 결과가 4, 5 쪽에 가까우면 `AltKey.Tools` 후보로 본다.

## 요약

이 문서의 결정은 다음 한 줄로 정리할 수 있다.

> AltKey는 설정 앱 전면 분리를 당장 추진하지 않고, 대신 한글 입력이 중요한 편집 기능을 별도 `AltKey.Tools` 앱으로 단계적으로 분리한다.
