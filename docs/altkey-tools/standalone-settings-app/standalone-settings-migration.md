# 설정 애플리케이션 독립 프로세스 분리 계획 (Standalone Settings Migration)

## 1. 개요 (Overview)
현재 `AltKey`의 설정 화면(`SettingsWindow`)은 메인 키보드 프로세스 내에 통합되어 있습니다. 이로 인해 자체 가상 키보드로 설정 항목을 입력할 때 동일 프로세스 내 포커스 전이 문제로 한글 자모가 분리되는 등 사용자 경험(UX) 상의 한계가 존재합니다. 이를 해결하기 위해 설정 기능을 별도의 실행 파일(`.exe`)로 분리하는 마이그레이션 계획을 수립합니다.

## 2. 필요성 및 목적
- **입력 안정성**: 별도 프로세스로 분리함으로써 가상 키보드가 설정 앱을 '외부 앱'으로 인식하게 하여 완벽한 한글 조합(Unicode/VirtualKey 방식 모두) 보장.
- **구조적 독립성**: UI 엔진과 설정 관리 로직의 결합도를 낮춰 유지보수 용이성 증대.
- **리소스 최적화**: 설정 창을 띄울 때만 리소스를 사용하고, 평상시에는 키보드 핵심 엔진만 가볍게 실행.

## 3. 제안 아키텍처 (Proposed Architecture)

### A. 프로젝트 구조 변경
- **AltKey.Core (Shared Library)**: 
    - `AppConfig`, `WindowConfig` 등 데이터 모델
    - `PathResolver`, `ConfigService` (공통 부분), `Logger` 등 유틸리티
- **AltKey.App (Main Keyboard)**: 가상 키보드 핵심 UI 및 입력 서비스
- **AltKey.Settings (Standalone App)**: 설정 UI 및 프로필 관리 전용 앱

### B. 프로세스 간 통신 (IPC) 및 동기화
설정 앱에서 값을 변경할 때 메인 앱에 즉시 반영하기 위한 전략:
1. **FileSystemWatcher**: `settings.json` 파일의 변경을 감지하여 메인 앱이 설정을 다시 로드 (가장 간편함).
2. **Named Pipes / Windows Message**: 변경 발생 시 즉각적인 신호를 전달하여 메모리 상의 설정을 실시간 동기화 (최고의 반응성).

## 4. 단계별 이관 계획 (Roadmap)

### 1단계: 공용 라이브러리 구축
- 두 앱이 공유해야 할 모델(`Models/`)과 핵심 서비스(`Services/ConfigService.cs` 등)를 별도 프로젝트로 추출합니다.

### 2단계: 설정 앱 프로젝트 생성
- WPF 프로젝트를 새로 생성하고 기존 `SettingsWindow.xaml` 및 `SettingsViewModel.cs`를 이관합니다.
- 새로운 앱의 진입점(`App.xaml.cs`)에서 필요한 서비스들을 의존성 주입(DI)으로 구성합니다.

### 3단계: 실행 및 인스턴스 제어
- 메인 앱의 트레이 메뉴 및 상단바 설정 버튼 클릭 시 `Process.Start`로 설정 앱을 실행하도록 수정합니다.
- Mutex 등을 이용해 설정 앱이 중복 실행되지 않고, 기존 창을 활성화하도록 처리합니다.

### 4단계: 실시간 동기화 구현
- IPC 채널을 구축하여 설정 앱의 변경 사항이 메인 앱의 UI(키보드 크기, 테마 등)에 즉시 반영되도록 구현합니다.

## 5. 기대 효과 및 고려 사항
- **효과**: 자체 가상 키보드를 이용한 설정 입력 시 발생하는 모든 버그(자모 분리 등) 해결.
- **주의**: 두 프로세스가 동시에 설정 파일에 접근할 때 발생할 수 있는 파일 잠금(File Locking) 문제를 방지하기 위해 원자적 쓰기(Atomic Write) 로직 강화 필요.

# 작성해주신 `AltKey`의 설정 애플리케이션 독립 프로세스 분리 계획[cite: 1]은 가상 키보드 개발 시 흔히 마주하는 IME(입력기) 포커스 충돌 문제를 해결하기 위한 가장 정석적이고 확실한 접근 방식입니다. 

자체 가상 키보드가 설정 앱을 '외부 앱'으로 인식하게 만들어 완벽한 한글 조합을 보장하겠다는 목적[cite: 1]은 이 아키텍처를 통해 완벽히 달성될 수 있습니다. 

계획하신 내용이 이미 훌륭하게 구조화되어 있으므로, 목표하시는 **'가볍고, 무설치로도 동작하며, 심미적으로 완성도 높은(애플과 같은) 소프트웨어'**라는 비전을 완성하기 위해 추가로 고려하면 좋을 아키텍처 및 UX 포인트를 몇 가지 제안해 드립니다.

### 1. 배포 및 '무설치(Portable)' 환경 유지 방안
프로세스를 두 개(`AltKey.App.exe`, `AltKey.Settings.exe`)로 분리하면[cite: 1], 프로그램 폴더가 무거워지거나 사용자 입장에서 파편화되어 보일 우려가 있습니다.
* **단일 파일 배포(Single-file deployment):** .NET 8/9 환경이라면 각각의 프로젝트를 단일 실행 파일로 퍼블리시하여, 사용자가 압축만 풀면 깔끔하게 두 개의 `.exe` 파일만 보이도록 구성하는 것이 좋습니다.
* **디렉토리 구조 숨기기:** 설정 앱 실행 파일을 `bin`이나 `core` 같은 하위 폴더에 배치하고, 메인 키보드 앱에서 상대 경로로 이를 호출하게 하면 사용자가 체감하는 구조를 훨씬 단순하게 유지할 수 있습니다.

### 2. IPC(프로세스 간 통신) 방식의 최적화
계획하신 실시간 동기화 구현[cite: 1]에서 두 가지 방식을 고민하셨는데, 안정성을 위해 다음과 같이 역할을 나누는 것을 추천합니다.
* **하이브리드 접근법:** 
    * **즉각적 UI 반영:** Named Pipes를 주력으로 사용하여 테마 변경, 키보드 크기 조절 등의 시각적 피드백이 딜레이 없이 애플 제품처럼 부드럽게 즉시 반영되도록 합니다[cite: 1].
    * **안전망(Fallback):** `FileSystemWatcher`는 백그라운드에 낮게 띄워두어, Named Pipe 통신이 모종의 이유로 끊겼을 때 파일 변경을 감지하여 재동기화하는 보조 수단으로 활용합니다[cite: 1]. (단, `FileSystemWatcher`는 파일 저장 시 이벤트가 여러 번 중복 발생(Bouncing)하는 고질적인 문제가 있으므로 디바운싱(Debouncing) 처리가 필수적입니다.)

### 3. 프로세스 생명주기(Lifecycle) 동기화 (고아 프로세스 방지)
설정 창을 띄울 때만 리소스를 사용하도록 분리하는 것[cite: 1]은 리소스 최적화에 탁월합니다. 하지만 예외 상황에 대한 처리가 필요합니다.
* **메인 앱 종료 시:** 메인 키보드 앱이 업데이트되거나 강제 종료될 때, 띄워져 있던 설정 앱이 있다면 함께 종료되도록 Named Pipe를 통해 '종료 시그널'을 보내야 합니다.
* **키보드 재시작 로직:** 설정 창에서 '초기화'나 '앱 재시작'이 필요한 중대한 설정을 건드렸을 때, 설정 앱이 메인 키보드 프로세스를 안전하게 죽이고 다시 띄울 수 있는 권한 및 로직이 필요합니다.

### 4. 시각적 단절감 해소 (심미성 및 UX)
프로세스가 분리되면 윈도우 OS 특성상 설정 창이 뜰 때 번쩍이거나 어색한 딜레이가 발생할 수 있습니다. 
* **애니메이션 및 포지셔닝:** 설정 앱이 켜질 때 윈도우 화면 중앙이나 트레이 아이콘 근처 등 일관된 위치에서 부드러운 페이드인(Fade-in) 효과와 함께 등장하도록 WPF 애니메이션을 적용하면, 독립된 앱이 아닌 하나의 유기적인 앱처럼 느껴집니다.
* **활성화 상태 동기화:** 3단계에서 언급하신 Mutex 제어 시[cite: 1], 이미 설정 창이 켜져 있을 때 사용자가 트레이에서 설정을 다시 누르면, 단순히 기존 창을 활성화(Activate)하는 것을 넘어 창이 살짝 흔들리거나 시각적 하이라이트를 주어 사용자의 인지 피로도를 낮춰주는 것이 좋습니다.

### 5. 원자적 쓰기(Atomic Write)와 설정 파일 무결성
말씀하신 파일 잠금(File Locking) 문제[cite: 1]는 실제로 매우 빈번하게 발생합니다.
* 설정을 저장할 때 `settings.json`에 직접 덮어쓰지 마시고, `settings.tmp`라는 임시 파일에 먼저 쓴 뒤, 쓰기가 완료되면 원본 파일을 대체(Replace)하는 **원자적 교체 연산(예: `File.Replace`)**을 사용하시기 바랍니다. 이렇게 하면 메인 프로세스가 설정 파일을 읽는 도중에 설정 앱이 파일을 쓰더라도 충돌이나 데이터 깨짐 현상을 원천 차단할 수 있습니다.

전체적인 로드맵과 1~4단계의 흐름이 매우 논리적이고 탄탄합니다. 공용 라이브러리(`AltKey.Core`)를 구축하여 데이터 모델을 공유하는 1단계[cite: 1] 작업이 성공적으로 마무리된다면, 나머지 마이그레이션은 수월하게 진행될 것으로 보입니다. 멋진 유틸리티의 탄생을 응원합니다!
